## Feature: Per-Tenant API Rate Limiting

### Context
The platform currently has no API usage guardrails. All tenants share the same infrastructure with no per-tenant limits. As the user base grows, a single high-traffic tenant can degrade performance for others. We need a rate limiting system that enforces per-tenant request quotas based on their subscription plan, returns standard rate limit headers, and blocks excess requests with a 429 response.

### Implementation Steps

**Step 1 — Add Prisma models**  
File: `prisma/schema.prisma`  
Add two models at the bottom of the file:

```prisma
model TenantPlan {
  id           String   @id @default(cuid())
  tenantId     String   @unique
  planName     String   @default("free")
  requestLimit Int      @default(1000)
  windowHours  Int      @default(1)
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt
}

model RateLimitLog {
  id           String   @id @default(cuid())
  tenantId     String
  windowStart  DateTime
  requestCount Int      @default(0)
  updatedAt    DateTime @updatedAt

  @@unique([tenantId, windowStart])
  @@index([tenantId])
}
```

**Step 2 — Run Prisma migration**  
Command: `npx prisma migrate dev --name add_rate_limit_tables`  
This generates the migration SQL and updates the Prisma client types. Confirm migration runs without error before proceeding.

**Step 3 — Create `src/lib/tenantPlan.ts`**  
Export one function: `getTenantLimit(tenantId: string): Promise<number>`  
- Query `TenantPlan` where `tenantId` matches
- If no row found → return `100` as default free-tier limit (do not throw)
- Return `plan.requestLimit`

```typescript
import { prisma } from "./db";

export async function getTenantLimit(tenantId: string): Promise<number> {
  const plan = await prisma.tenantPlan.findUnique({
    where: { tenantId },
  });
  return plan?.requestLimit ?? 100;
}
```

**Step 4 — Create `src/lib/rateLimit.ts`**  
Export two functions: `checkRateLimit()` and `incrementUsage()`.

`getWindowStart()` helper: truncate current UTC time to the current hour.

```typescript
function getWindowStart(): Date {
  const now = new Date();
  now.setMinutes(0, 0, 0);
  return now;
}
```

`checkRateLimit(tenantId: string): Promise<RateLimitResult>`  
- Call `getTenantLimit(tenantId)` to get the limit
- Query `RateLimitLog` by `{ tenantId, windowStart }`
- Return `{ allowed, remaining, resetAt, limit }` — never throw

`incrementUsage(tenantId: string): Promise<void>`  
- Upsert `RateLimitLog` using `{ increment: 1 }` on `requestCount`
- Use `@@unique([tenantId, windowStart])` as the upsert key
- Never read-then-write — always use upsert+increment

Type definition to export:
```typescript
export type RateLimitResult = {
  allowed: boolean;
  remaining: number;
  resetAt: Date;
  limit: number;
};
```

**Step 5 — Modify `src/middleware.ts`**  
- Extract `tenantId` from the request header `x-tenant-id`
- If header is missing → return 400 with `{ error: "Missing x-tenant-id header" }`
- Call `checkRateLimit(tenantId)`
- If `result.allowed === false` → return `NextResponse.json(...)` with status 429
  - Body: `{ error: "Rate limit exceeded", retryAfter: <seconds>, resetAt: <ISO string> }`
  - Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`
- If allowed → call `incrementUsage(tenantId)`, then `NextResponse.next()` with rate limit headers attached

**Step 6 — Modify `src/app/api/data/route.ts`**  
- This route's actual business logic is NOT changing
- Add rate limit headers to the existing success response:
  - `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- To get the values: call `checkRateLimit(tenantId)` at the top of the handler (read-only — do not call `incrementUsage()` here, middleware already did)
- Extract `tenantId` from `request.headers.get("x-tenant-id")`
- If header missing → return 400 (same as middleware, defensive)

### Expected Behavior

| Scenario | Expected |
|---|---|
| Request within limit | 200 with rate limit headers |
| Request exceeds limit | 429 with `Retry-After` and reset time |
| Missing `x-tenant-id` header | 400 with error message |
| Tenant has no plan row | Apply 100 req/hour default, do not error |
| Concurrent requests at limit boundary | Upsert+increment ensures no double-counting |

**Edge cases:**
- Window resets at the top of each hour (not rolling — fixed window)
- `requestCount` at exactly the limit → next request is rejected (limit is exclusive upper bound)
- DB unreachable → fail open (allow request), log error with `console.error`
